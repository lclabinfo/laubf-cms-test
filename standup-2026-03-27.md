## Key Concepts (for reference / explaining to others)

### ISR (Incremental Static Regeneration)
ISR is Next.js's way of caching entire pages at the server level. When a user visits a page, Next.js serves a **static HTML snapshot** instead of running the React component + database queries from scratch every time. You control how long the cache lives with `export const revalidate = 60` (seconds) in the route file.

**How it works:**
1. First visitor hits `/messages` → Next.js runs the full render (DB queries, React, HTML), serves the result, and **saves a copy** on disk.
2. Next visitor within 60s → gets the saved copy instantly. Zero DB queries, zero React rendering.
3. After 60s, the next visitor still gets the stale copy (fast!), but Next.js **regenerates a fresh version in the background**. The *following* visitor gets the new one.
4. When a CMS user publishes a new message, the API route calls `revalidatePath('/messages')` which **immediately invalidates** the cached page — the next visitor triggers a fresh render.

**Why it matters:** Our website pages were hitting the database on every single page view. With ISR, a page that gets 100 visits/minute only runs 1 DB query/minute (or fewer). It's the single biggest performance win for the least code change.

**Limitation:** ISR caches the *entire page*. If two different pages share the same DAL query (e.g., both the messages page and the homepage call `getLatestMessage()`), ISR can't share that cached query between them — each page caches independently.

### `unstable_cache` (Query-Level Caching)
Where ISR caches whole pages, `unstable_cache` caches **individual function calls** (like a single database query). You wrap a DAL function and give it a cache key + tags:

```ts
const cachedGetMessages = unstable_cache(
  (churchId, page, search) => getMessages(churchId, { page, search }),
  ['messages', churchId, page, search],  // cache key
  { tags: [`church:${churchId}:messages`], revalidate: 300 }
)
```

**How it works:**
1. First call with `(churchId, page=1, search="")` → runs the real DB query, caches the JSON result.
2. Same call within 300s → returns the cached JSON. No DB hit.
3. CMS user creates a message → API route calls `revalidateTag('church:xyz:messages')` → all cached message queries for that church are invalidated.

**Why we added it on top of ISR:** ISR only helps the public website (server-rendered pages). CMS pages are fully dynamic (they call `headers()` for auth), so ISR can't cache them. `unstable_cache` caches the underlying queries regardless of whether the page itself is static or dynamic. It also lets two different pages that call `getMessages()` share the same cached result.

**Gotcha we hit:** `unstable_cache` serializes everything to JSON internally. This silently turns `Date` objects into strings like `"2026-03-15T00:00:00.000Z"`. Any code that calls `.toISOString()` on the result crashes because strings don't have that method. We had to add `instanceof Date` guards in 6 places.

### `use cache` (Next.js 16 Native — Not Yet Implemented)
The "official" successor to `unstable_cache`. You just add `'use cache'` at the top of a function (like `'use server'`). More ergonomic, better integrated with React's streaming model.

**Why we didn't use it:** It requires the `cacheComponents` flag in `next.config.ts`, which makes `headers()` and `cookies()` calls throw errors unless wrapped in `<Suspense>`. Seven of our CMS pages call `headers()` for auth and would need refactoring. Since ISR + `unstable_cache` already covers 95% of the benefit, we documented the plan and deferred it.

### How They Layer Together
```
Layer 1: ISR (page-level)     → Public website pages cached as static HTML
Layer 2: unstable_cache        → Individual DB queries cached in memory
Layer 3: Prisma query engine   → Connection pooling, prepared statements
Layer 4: PostgreSQL            → Partial indexes, TOAST optimization, LZ4
```
Each layer reduces load on the layer below it. A cached ISR page never even calls `unstable_cache`. But when ISR misses (stale or dynamic page), `unstable_cache` prevents redundant DB queries. And when the DB does get hit, the PostgreSQL tuning makes those queries faster.

## Performance & Memory Optimization

### Memory Audit & Investigation
Ran a 6-agent investigation across the entire codebase to understand memory usage patterns. The biggest finding: **list queries were loading full text/JSON content the UI never displays**. `getBibleStudies()` alone was transferring ~80 MB for 1,185 studies when the list page only needs ~152 KB of metadata — a **500x over-fetch**.

### Selective DAL Queries (Priority 1 from audit)
Attacked the over-fetching problem at multiple layers:
- **Per-query `omit`** on list endpoints — strip transcript, questions, answers, bibleText from list responses
- **Global `omit` on PrismaClient** as a safety net — heavy text fields excluded by default, detail queries opt back in explicitly
- **`take` limits on nested includes** — bounded attachments, role assignments, household members (previously unbounded)
- **React `cache()` deduplication** — detail pages that call the same DAL function in both `generateMetadata()` and the page component now share one query instead of two

This caused a regression: the global omit accidentally excluded BibleStudy questions/answers from the message detail view, making 1,152 Bible study tabs show empty. Fixed by adding an explicit opt-in on the nested `relatedStudy` include.

### PostgreSQL Tuning (Priority 6 from audit)
Free performance wins requiring no application code changes:
- **Partial indexes** on Message/Event for `WHERE deletedAt IS NULL` — almost every query filters soft-deleted rows, but existing indexes included them all
- **LZ4 compression** on 6 TOAST columns — 3-5x faster decompression than default pglz for transcript and description fields
- **STORAGE MAIN** on `PageSection.content` — keeps small (<2 KB) but frequently-read JSON inline instead of in the TOAST table, avoiding an extra I/O hop on every page render
- **Dropped 9 redundant indexes** — boolean columns on small tables, duplicates of `@unique` auto-indexes

### ISR + API Guards + Builder Caps
- **ISR** (biggest caching win) — added `export const revalidate` to all `app/website/` routes (60-3600s), set `cacheMaxMemorySize: 50MB`. Eliminates DB queries entirely for cached website pages. Existing `revalidatePath()` calls in CMS API routes handle invalidation.
- **API guards** — capped `pageSize` at 100 on people/events/messages/bible-studies/videos APIs, added `ids.length > 1000` guard on bulk endpoints. This broke the members page which was requesting 500 at once — temporarily raised to 500 with a TODO to refactor to server-side pagination (done later same day).
- **Builder memory** — undo history capped at 10 entries (was 50, ~2 MB saved), pages query bounded with `take: 100`
- **Rate limiter** — cleanup interval 5min → 1min, max entries 10K → 5K, added inline expiry check

## Server-Side Pagination & Filtering

### The Pattern
The codebase had a recurring anti-pattern: load the entire dataset into browser memory, then do client-side search/filter/pagination. The messages page was making 24+ background API calls on mount to eager-fetch all pages. This doesn't scale.

Established the "messages pattern" as the gold standard: server-side pagination (50 per page), debounced server-side search, server-side sorting, and `fetchIdRef` to prevent stale responses from overwriting newer results.

### Members Page Refactor
The CMS members page was loading all 500 members into React state. Refactored to match the messages pattern:
- 500 → 50 per page, server-driven pagination controls
- Server-side search with 300ms debounce
- Server-side sorting (name, email, createdAt, membershipStatus)
- Status filter wired to server-side context instead of client-side TanStack column filter
- **Memory impact:** ~2.5-5 MB per CMS session eliminated

### Website Listing Sections
All 4 public listing sections (bible studies, messages, events, videos) converted from eager-fetch-all to server-side:
- Filter changes trigger single API calls with params (dateFrom, dateTo, search, series, speaker, book, ministry, campus, sort)
- AbortController cancels in-flight requests when filters change rapidly
- Enhanced DAL functions with new filter support: `seriesName`, `speakerName` (resolves display name → person IDs), `ministryName`, `campusName`, and date ranges

### Filter Dropdown Completeness
Filter dropdowns (year, series, book, speaker) were derived from only the current page of results, showing incomplete options. Added lightweight server-side `filterMeta` queries (`groupBy`/`distinct`) that return complete filter options on first render, independent of pagination.

### Smooth Filter Transitions
Eliminated the jarring flash-of-empty-content when filters change:
- Old content stays visible and fades to 30% opacity (500ms ease-in-out)
- API fetches in background, 150ms pause before swapping
- Content fades back to 100% with new results
- `min-height` container preserves scroll position

## Caching Strategy & Evolution

### The Journey: ISR → `use cache` → `unstable_cache`
The caching approach evolved through the day as we hit real constraints:

1. **ISR (Incremental Static Regeneration)** — Implemented first. Biggest win for the least effort. Website pages get a static cache with time-based revalidation. CMS write operations call `revalidatePath()` to bust the cache. This handles ~90% of the caching benefit.

2. **`use cache` (Next.js 16 native)** — Investigated as the next step for query-level caching. Hit a blocker: `cacheComponents` flag is incompatible with CMS pages that call `headers()` (a dynamic API). Would require wrapping 7 CMS files in Suspense boundaries. Documented the plan but deprioritized — ISR already covers the main use case.

3. **`unstable_cache` (the middle ground)** — After reading the official Next.js 16.2.1 docs, discovered this works NOW without the Suspense refactor. Available since Next.js 14.1, stable enough for production. Decided this was the right next step.

### `unstable_cache` Implementation
Wrapped 24 read-only DAL functions across 10 files:
- Cache keys include `churchId` + all query parameters for proper multi-tenant isolation
- Tags use `church:{churchId}:{category}` for granular invalidation
- TTLs: 300s for list queries, 3600s for detail/static queries
- Mutation functions intentionally NOT cached

Wired tag-based invalidation into all 17 CMS write API routes — placed after successful mutations alongside existing `revalidatePath()` calls, so both ISR page cache AND query cache get cleared together.

### Date Serialization Gotcha
`unstable_cache` serializes responses to JSON under the hood, which silently converts `Date` objects into strings like `"2026-03-15T00:00:00.000Z"`. Any downstream code calling `.toISOString()` or `.getTime()` on what it assumes is a Date object crashes at runtime. Fixed 6 locations with `instanceof Date` guards and `new Date()` wrappers for Date|string compatibility.

## Bug Fixes
- **Media upload with non-ASCII filenames** — macOS screenshots include an invisible narrow no-break space (U+202F) between time and AM/PM. This caused `ERR_INVALID_CHAR` when setting the `Content-Disposition` header during staging→permanent promotion in R2. Fixed by stripping all non-printable-ASCII characters while preserving the UTF-8 filename via RFC 5987's `filename*` parameter.
- **Incomplete filter dropdowns** — Filters derived from page 1 of data showed incomplete options. Added server-side `filterMeta` queries.
- **Messages page missing 27 videos** — Hard cap of 200 meant messages 201-227 were invisible. Switched to paginated with eager-fetch of remaining pages (later replaced by full server-side pagination).
- **Bible Study tab empty** — Global `omit` from the memory optimization excluded `questions`/`answers` from the nested `relatedStudy` include. 1,152 of 1,160 entries affected.
- **Empty state wrong after pagination** — Was checking `members.length` (current page, max 50) instead of `pagination.total` (full dataset).
- **7 date timezone bugs** — `getFullYear()` → `getUTCFullYear()`, `setHours(0,0,0,0)` → UTC midnight, `toLocaleDateString()` on raw Prisma Dates converted through ISO string split. Fixed a ghost "1999" year appearing in filter dropdowns (caused by Jan 1 2000 UTC dates rolling back to Dec 31 1999 in PST).
