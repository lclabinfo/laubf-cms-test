## 1. URL Routing Architecture (Proxy vs Rewrites)

**The problem:** We had two competing systems for subdomain routing — `proxy.ts` (Next.js 16 convention) AND `next.config.ts` beforeFiles rewrites. On production, both fired, causing double-rewriting (`/website/website/...`).

**What we did:**
- Started by adding `next.config.ts` rewrites (`19137c0`) as a first attempt
- Discovered the conflict with `proxy.ts`, reversed course (`c7ab93a`)
- Removed the config rewrites entirely — `proxy.ts` is the single source of truth
- Wrote `docs/04_proxy-routing/proxy-routing-architecture.md` documenting the full request flow and decision rationale
- Added `NEXT_PUBLIC_SITE_URL` to `.env.example` for canonical URL generation

**Key insight:** `proxy.ts` is the correct Next.js 16 approach. Config-level rewrites are the old pattern and caused the double-prefix bug on prod.

## 5. Attachment Downloads (R2 Cross-Origin)

This was a two-commit rabbit hole (`7d8d2cd` → `6daed1d`):

**The problem:** R2's public URLs (`r2.dev`) don't expose `Content-Disposition` headers via CORS. `curl` sees the header, but browsers don't. So the `<a href download>` attribute was ignored — files opened in a new tab with UUID filenames instead of downloading with original names.

**The fix:** Blob download approach.
```
fetch(r2Url) → response.blob() → URL.createObjectURL(blob) → <a download="original-name.pdf">
```
Blob URLs are same-origin, so `download` attribute is honored. Applied to both website bible study attachments and CMS attachment downloads.

**Bonus:** Rewrote the R2 backfill script to use `CopyObject` (server-side metadata copy, zero data transfer) instead of download+reupload. Content-Disposition headers on R2 objects are still set for custom domains and non-browser clients.

## 6. Builder Nested Slug Fix

Quick but impactful (`fb465d8`):

Pages with nested slugs like `ministries/campus/usc` broke the builder save because `pagePathId()` used the raw slug for API calls. Slashes created multi-segment URLs that didn't match the `[slug]` dynamic route → PATCH returned 404 silently → post-save refetch also 404'd → builder thought sections were "deleted by another user" and dropped them from preview.

**Fix:** Use page UUID instead of slug when the slug contains slashes.
